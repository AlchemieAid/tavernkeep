# Windsurf Project Prompt: TavernKeep — D&D Shop Sharing Platform

## Project Overview

Build **TavernKeep**, a full-stack mobile-first web application that lets Dungeon Masters (DMs) create, manage, and share fantasy shops with their D&D players. Players join via QR code and browse a living, AI-enhanced storefront experience. The DM has a secure profile-based dashboard for shop management; players have a read-only browsing experience.

## Advanced Feature Specs

### Party View / Campaign Mode
- Multiple shops per campaign; only one can have `is_active = true` at a time
- Activating one shop deactivates all others in the same campaign automatically
- Campaign-level QR code always resolves to the currently active shop

### Inventory Volatility Engine
- `slow` ≈ 10% item rotation per restock, `moderate` ≈ 30%, `fast` ≈ 60%
- Rotation: set `expires_at` on items; generate replacements via GPT-4o using shop context
- Expired items are soft-deleted (`deleted_at` timestamp) — never hard deleted
- DM can view archived item history per shop

### AI Item Generation (GPT-4o)
Prompt with: shop type, economic tier, location descriptor, existing item count and categories.
Return structured JSON only: `name`, `description`, `category`, `rarity`, `base_price_gp`, `properties`.

### AI Shopkeeper Generation (GPT-4o)
Prompt with: race, name (or generate one), shop type.
Return JSON: `backstory`, `motivation`, `personality_traits[]`, `player_greeting`.

### AI Image Generation (DALL-E 3)
- **Keeper portrait**: race + personality traits + shop type → painterly fantasy portrait style
- **Exterior**: shop type + economic tier + location → fantasy establishing shot
- **Interior**: shop type + tier + item categories → warm candlelit interior

Store all generated images in Supabase Storage. Never regenerate unless explicitly requested by the DM.

### Hidden Items
- `is_hidden = true` items stripped at the API level — never reach the player UI
- DM toggles visibility per item in the shop editor
- `hidden_condition` stores DM-facing notes for reveal trigger
- When DM reveals an item, it appears in the player view in realtime

### Price Control
- Player sees: `Math.round(base_price_gp * price_modifier)` gp
- DM editor shows both the base price and the effective price side by side
- `price_modifier` slider ranges from 0.5 (50%) to 2.0 (200%)

### Haggling
- When `haggle_enabled = true`, player view shows the `haggle_dc` and a flavor prompt
- Player rolls a Persuasion check physically, reports result to DM
- DM manually adjusts `price_modifier` if the check succeeds

## New Requirements 2026-04-06: Live Multiplayer Campaign System

### Vision
Transform TavernKeep from a static shop-sharing tool into a **live multiplayer campaign explorer** where DMs and players collaborate in real-time during game sessions.

### Core Concept: Two Player Modes

**1. Static Explorer Mode** (Default)
- Players browse all revealed content at their own pace
- Navigate freely through campaigns, towns, shops, people, items
- Shopping cart persists across sessions
- No real-time sync required

**2. Live Session Mode** (DM-controlled)
- DM "broadcasts" their current view to all players
- Players see exactly what the DM is viewing (minus hidden content)
- Real-time sync via Supabase Realtime
- DM can toggle players in/out of live mode with one button
- Players can still interact (add to cart) while in live mode

---

### Architectural Decisions

#### Player Authentication Strategy
**Decision**: Use Supabase Auth with multiple providers
- Google OAuth (existing)
- Magic link email auth (passwordless, easier for players)
- Anonymous guest mode with session persistence (optional future)

**Rationale**: Players need persistent identity for inventory/cart, but shouldn't require complex signup.

#### Campaign Membership Model
**Decision**: Invitation-based with QR codes and shareable links
- DM generates campaign invite link/QR code
- Players scan/click to join campaign
- Membership persists across sessions
- DM can revoke access

**Rationale**: Simple onboarding, secure access control, familiar pattern.

#### Visibility System Architecture
**Decision**: Hierarchical visibility with `is_revealed` boolean
- **Campaigns**: Always visible to members (no visibility toggle)
- **Towns**: `is_revealed` boolean (default: false)
- **Shops**: `is_revealed` boolean (default: false)
- **Notable People**: `is_revealed` boolean (default: false)
- **Items**: Existing `is_hidden` + `reveal_state` system (unchanged)

**Rationale**: 
- Consistent pattern across hierarchy
- Items already have sophisticated visibility (keep it)
- Campaigns are invite-only so no need for reveal toggle

#### Real-Time Sync Architecture
**Decision**: Supabase Realtime with selective subscriptions
- **Live Mode**: Subscribe to `dm_sessions` table for DM's current view
- **Static Mode**: No subscriptions (reduces load)
- **Shopping Cart**: Subscribe to `cart_items` for multi-player cart visibility
- **Inventory**: No real-time sync needed (updates on transaction finalize)

**Rationale**:
- Supabase Realtime is already in stack
- Selective subscriptions minimize database load
- Cart sync is critical for collaborative shopping
- Live mode is opt-in, not always-on

#### Shopping Cart Design
**Decision**: Shared cart per player within a campaign
- Each player has ONE cart per campaign
- Cart items reference shop_id + item_id + quantity
- All players + DM can see all carts simultaneously
- DM finalizes transactions (moves cart → inventory)

**Rationale**:
- Simple mental model (one cart per campaign)
- Collaborative visibility builds trust
- DM has final approval (prevents exploits)

#### Inventory System Design
**Decision**: Player-owned inventory with item state tracking
- Items copied from shop to player inventory on transaction
- Snapshot of all item properties at purchase time
- State field: `available`, `equipped`, `used`, `consumed`, `stolen`, `broken`, `gifted`, `lost`, `sold`
- Filterable by state in player UI
- Items removed from shop stock on finalize

**Rationale**:
- Preserves item properties at purchase (e.g., price, stats)
- Flexible state system supports various gameplay scenarios
- Immutable purchase history (items never deleted, only state changes)

### Database Schema Changes

#### New Table: `players`
```sql
CREATE TABLE players (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name TEXT NOT NULL,
  avatar_url TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

#### New Table: `campaign_members`
```sql
CREATE TABLE campaign_members (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  campaign_id UUID REFERENCES campaigns(id) ON DELETE CASCADE,
  player_id UUID REFERENCES players(id) ON DELETE CASCADE,
  invited_by UUID REFERENCES profiles(id),
  joined_at TIMESTAMPTZ DEFAULT now(),
  last_active_at TIMESTAMPTZ DEFAULT now(),
  is_active BOOLEAN DEFAULT true,
  UNIQUE(campaign_id, player_id)
);
```

#### New Table: `dm_sessions`
```sql
CREATE TABLE dm_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  campaign_id UUID REFERENCES campaigns(id) ON DELETE CASCADE,
  dm_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  current_view_type TEXT,
  current_view_id UUID,
  live_mode_enabled BOOLEAN DEFAULT false,
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(campaign_id)
);
```

#### New Table: `cart_items`
```sql
CREATE TABLE cart_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  campaign_id UUID REFERENCES campaigns(id) ON DELETE CASCADE,
  player_id UUID REFERENCES players(id) ON DELETE CASCADE,
  shop_id UUID REFERENCES shops(id) ON DELETE CASCADE,
  item_id UUID REFERENCES items(id) ON DELETE CASCADE,
  quantity INT NOT NULL DEFAULT 1 CHECK (quantity > 0),
  added_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(player_id, campaign_id, item_id)
);
```

#### New Table: `player_inventory`
```sql
CREATE TABLE player_inventory (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  player_id UUID REFERENCES players(id) ON DELETE CASCADE,
  campaign_id UUID REFERENCES campaigns(id) ON DELETE CASCADE,
  item_name TEXT NOT NULL,
  item_description TEXT,
  item_category TEXT,
  item_rarity TEXT,
  purchase_price_gp INT NOT NULL,
  item_weight_lbs FLOAT,
  item_properties JSONB,
  item_image_url TEXT,
  shop_id UUID REFERENCES shops(id) ON DELETE SET NULL,
  shop_name TEXT,
  purchased_at TIMESTAMPTZ DEFAULT now(),
  state TEXT DEFAULT 'available' CHECK (state IN (
    'available', 'equipped', 'used', 'consumed',
    'stolen', 'broken', 'gifted', 'lost', 'sold'
  )),
  state_notes TEXT,
  state_changed_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

#### New Table: `transactions`
```sql
CREATE TABLE transactions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  campaign_id UUID REFERENCES campaigns(id) ON DELETE CASCADE,
  shop_id UUID REFERENCES shops(id) ON DELETE SET NULL,
  player_id UUID REFERENCES players(id) ON DELETE CASCADE,
  dm_id UUID REFERENCES profiles(id) ON DELETE SET NULL,
  total_price_gp INT NOT NULL,
  item_count INT NOT NULL,
  items_snapshot JSONB NOT NULL,
  finalized_at TIMESTAMPTZ DEFAULT now(),
  notes TEXT
);
```
