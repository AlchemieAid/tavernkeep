# Windsurf Project Prompt: TavernKeep — D&D Shop Sharing Platform

## Project Overview

Build **TavernKeep**, a full-stack mobile-first web application that lets Dungeon Masters (DMs) create, manage, and share fantasy shops with their D&D players. Players join via QR code and browse a living, AI-enhanced storefront experience. The DM has a secure profile-based dashboard for shop management; players have a read-only browsing experience.

## Database Schema

Design a Supabase PostgreSQL schema with the following tables. **Use only the Supabase anon key for all operations** — there is no service role key at this stage. All access control must be enforced through RLS policies.

### `profiles`
- `id` (uuid, FK → auth.users)
- `display_name` (text)
- `avatar_url` (text)
- `created_at` (timestamp)

### `campaigns`
- `id` (uuid)
- `dm_id` (uuid, FK → profiles)
- `name` (text)
- `description` (text)
- `created_at` (timestamp)

### `shops`
- `id` (uuid)
- `campaign_id` (uuid, FK → campaigns)
- `dm_id` (uuid, FK → profiles)
- `name` (text)
- `slug` (text, unique) — used in QR code URLs, generated with nanoid (10 chars)
- `is_active` (boolean) — controls party visibility
- `shop_type` (enum: `general`, `weapons`, `armor`, `magic`, `apothecary`, `black_market`)
- `location_descriptor` (text) — e.g., "dockside port town"
- `economic_tier` (enum: `poor`, `modest`, `comfortable`, `wealthy`, `opulent`)
- `price_modifier` (float, default 1.0) — DM-controlled 0.5 to 2.0
- `haggle_enabled` (boolean)
- `haggle_dc` (int) — Difficulty Class for haggling check
- `inventory_volatility` (enum: `static`, `slow`, `moderate`, `fast`)
- `last_restocked_at` (timestamp)
- `keeper_name` (text)
- `keeper_race` (text)
- `keeper_backstory` (text)
- `keeper_motivation` (text)
- `keeper_personality_traits` (text[])
- `keeper_image_url` (text)
- `shop_exterior_image_url` (text)
- `shop_interior_image_url` (text)
- `created_at` (timestamp)
- `updated_at` (timestamp)

### `items`
- `id` (uuid)
- `shop_id` (uuid, FK → shops)
- `name` (text)
- `description` (text)
- `category` (enum: `weapon`, `armor`, `potion`, `scroll`, `tool`, `magic_item`, `misc`)
- `rarity` (enum: `common`, `uncommon`, `rare`, `very_rare`, `legendary`)
- `base_price_gp` (int)
- `stock_quantity` (int)
- `is_hidden` (boolean, default false)
- `hidden_condition` (text) — DM notes for reveal trigger
- `image_url` (text, nullable)
- `weight_lbs` (float)
- `properties` (jsonb) — flexible weapon/armor stats
- `added_at` (timestamp)
- `expires_at` (timestamp, nullable) — for volatile inventory
- `deleted_at` (timestamp, nullable) — soft delete only, never hard delete

### `party_access`
- `id` (uuid)
- `campaign_id` (uuid, FK → campaigns)
- `player_alias` (text, nullable)
- `session_token` (text)
- `last_seen_at` (timestamp)

## Core Feature Specs

### DM Dashboard
- List all campaigns with shop counts
- Quick-toggle shop active/inactive per campaign
- Visual indicator for which shop the party currently sees

### Shop Creation Wizard (multi-step)
1. **Basic Info**: Name, type, location descriptor, economic tier
2. **Shopkeeper**: Name, race, backstory, motivation, personality traits
3. **Variables**: Price modifier slider (50%–200%), haggling toggle + DC, inventory volatility
4. **Items**: Add items manually or trigger AI generation
5. **Images**: Generate keeper portrait, exterior, interior via DALL-E 3, or skip for now
6. **Review + Activate**: Preview card + share QR code

### Shop Editor
- Full inline editing of all shop fields
- Toggle individual item visibility (show/hide from players)
- Hidden item conditions editor (DM-only notes)
- "Restock" button to trigger inventory volatility simulation
- Real-time preview panel of the player view

### QR Code Share Page (`/dm/shops/[shopId]/qr`)
- Large centered QR code linking to `/shop/[slug]`
- One-tap copy URL to clipboard
- Web Share API button for native mobile share sheet
- Clean fullscreen layout — DM flips their phone toward the party

### Player Shop View (`/shop/[slug]`)
- No login required — fully public route
- Mobile-first single-column card browser
- Shows: item name, description, modified price, rarity badge, stock
- Shopkeeper portrait + name + flavor greeting at top
- Exterior/interior image gallery
- Hidden items never appear — stripped at the API route level, not the UI
- Haggling prompt shown when enabled (displays DC, player rolls physically)
- Realtime subscription: player view updates live when DM makes changes

## Advanced Feature Specs

### Party View / Campaign Mode
- Multiple shops per campaign; only one can have `is_active = true` at a time
- Activating one shop deactivates all others in the same campaign automatically
- Campaign-level QR code always resolves to the currently active shop

### Inventory Volatility Engine
- `slow` ≈ 10% item rotation per restock, `moderate` ≈ 30%, `fast` ≈ 60%
- Rotation: set `expires_at` on items; generate replacements via GPT-4o using shop context
- Expired items are soft-deleted (`deleted_at` timestamp) — never hard deleted
- DM can view archived item history per shop

### AI Item Generation (GPT-4o)
Prompt with: shop type, economic tier, location descriptor, existing item count and categories.
Return structured JSON only: `name`, `description`, `category`, `rarity`, `base_price_gp`, `properties`.

### AI Shopkeeper Generation (GPT-4o)
Prompt with: race, name (or generate one), shop type.
Return JSON: `backstory`, `motivation`, `personality_traits[]`, `player_greeting`.

### AI Image Generation (DALL-E 3)
- **Keeper portrait**: race + personality traits + shop type → painterly fantasy portrait style
- **Exterior**: shop type + economic tier + location → fantasy establishing shot
- **Interior**: shop type + tier + item categories → warm candlelit interior

Store all generated images in Supabase Storage. Never regenerate unless explicitly requested by the DM.

### Hidden Items
- `is_hidden = true` items stripped at the API level — never reach the player UI
- DM toggles visibility per item in the shop editor
- `hidden_condition` stores DM-facing notes for reveal trigger
- When DM reveals an item, it appears in the player view in realtime

### Price Control
- Player sees: `Math.round(base_price_gp * price_modifier)` gp
- DM editor shows both the base price and the effective price side by side
- `price_modifier` slider ranges from 0.5 (50%) to 2.0 (200%)

### Haggling
- When `haggle_enabled = true`, player view shows the `haggle_dc` and a flavor prompt
- Player rolls a Persuasion check physically, reports result to DM
- DM manually adjusts `price_modifier` if the check succeeds

## Security Requirements

- RLS enabled on ALL Supabase tables — policies written before any data access code
- Only the anon key is used — all access control via RLS policies, no service role key
- Player API routes strip `is_hidden = true` items and all DM-only fields before returning
- DM write routes verify session `user.id` ownership before any mutation
- QR slugs generated with nanoid (10 chars)
- Rate limiting on AI endpoints: 5 requests/min per IP
- All API keys in environment variables only — never in client-side code

## New Requirements 2026-04-05

I want to add dropdown parameters to the different hierarchies that inform the system prompts behind ai-generated content. 
For Campaigns, I want the DM to be able to set the ruleset (5e, 4e, 3e, 2e, Pathfinder, Traveler, etc.), setting, history, currency, pantheon, and anything else that might be global variables.
For Towns, I want the DM to be able to set population, geographic size, geographic location (dessert, forest, wilderness, necropolis, arctic, plains, riverside, etc.), ruler, political system, history, notable figures, and anything else that might be at the city level.
Speaking of notable people, we should turn Shop Keepers into Notable People and then make some of the Notable People run Shops. Other Notable people can be quest givers, rulers, priests, magicians, etc. Each Town can be populated with however many notable people the DM desires. Notable people should additionally have a drop-down menu at the top just like Shops.
For Shops, I think you have a good number of parameters already set, but analyze the future state and determine if other variables are needed. Add the ability to assign Shop Keepers from the Notable People that live in each town.
For Items, I also think you have a good number of parameters already established, but analyze them for additional requirements set by the future state need.
When generating with AI, the AI should automatically select the proper settings for Campaigns, Towns, Notable People, Shops, and Items.
The AI should start a “Wizard” when invoked at the campaign level where the DM could specify the number of towns needed and the AI can automatically invoke Shops, Notable People, Shop Keeper Assignments, and Items based on the hierarchical structure defined here.
