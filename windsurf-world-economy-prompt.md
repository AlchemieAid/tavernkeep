# Windsurf Feature Prompt: Campaign World Economy, Map System, PoI & Dual-Path Creation

## Branch

```bash
git checkout -b feature/campaign-world-economy
```

Do not merge to main until: a DM can create or upload a map, see the resource overlay, place Points of Interest, hover/tap to inspect resource levels, and place a town with economically derived characteristics.

---

## Core Design Philosophy

**No grid cells. No discrete coordinates.**

The world is a continuous 2D surface. Coordinates are normalized floats (0.0–1.0, percentage of image dimensions). Resources exist as points and terrain polygons. At any (x, y), resource values are computed in real time via spatial interpolation. This is smooth, natural, and has no grid artifacts.

**Two map creation paths exist and are equally supported.**

- **AI Path**: DM configures parameters → DALL-E 3 generates 3 map options → DM selects one → AI classifies terrain and places resources automatically
- **Upload Path**: DM uploads their own map image → AI assists with terrain classification and resource placement, or DM places everything manually

Both paths produce identical data structures. All downstream features (resource hover, town placement, PoI, shops) work identically regardless of which path created the map. Design every page and component with both paths in mind from the start.

---

## What AI Does (narrow and bounded)

1. **DALL-E 3**: Generates 3 candidate map images (AI path only)
2. **GPT-4o Vision**: Classifies terrain areas as polygons (both paths — runs on whichever image exists)
3. **GPT-4o**: Places resource points based on terrain analysis (both paths — DM can override)
4. **GPT-4o**: Generates town prose using computed economic context (both paths)

AI never computes resource values, wealth scores, price indexes, or town tiers.

## Database Schema

Write all migrations and RLS policies before any application code.

### `campaign_maps` (updated)
```sql
id                uuid primary key default uuid_generate_v4()
campaign_id       uuid references campaigns(id) on delete cascade
dm_id             uuid references profiles(id)
image_url         text not null
is_selected       boolean default false
width_px          int
height_px         int
map_size          text not null
  -- 'region'    ~200km²   → 30–60 resource points
  -- 'kingdom'   ~2000km²  → 80–140 resource points
  -- 'continent' ~20000km² → 200–350 resource points
map_style         text   -- 'fantasy_painted','parchment','hand_drawn','topographic'
biome_profile     text   -- 'temperate','arctic','tropical','arid','archipelago','volcanic'
creation_method   text not null default 'ai'
  -- 'ai'       → generated by DALL-E 3
  -- 'uploaded' → provided by DM
-- AI generation tracking (only relevant when creation_method = 'ai')
generation_count  int default 1
-- Upload metadata (only relevant when creation_method = 'uploaded')
original_filename text
-- Workflow state — tracks where in setup the map currently is
setup_stage text not null default 'created'
  -- 'created'              → map image exists, nothing else done yet
  -- 'terrain_classified'   → terrain polygons written
  -- 'resources_placed'     → resource points written
  -- 'ready'                → fully set up, DM can place towns and PoIs
created_at        timestamptz default now()
```

### `terrain_areas`
```sql
id             uuid primary key default uuid_generate_v4()
map_id         uuid references campaign_maps(id) on delete cascade
terrain_type   text not null
  -- 'ocean','deep_sea','coast','river','lake',
  -- 'plains','grassland','farmland',
  -- 'forest','deep_forest','jungle',
  -- 'hills','highlands','mountains','high_mountains',
  -- 'swamp','wetlands','desert','badlands',
  -- 'tundra','arctic','volcanic'
polygon        jsonb not null   -- [{"x": 0.12, "y": 0.34}, ...]
placed_by      text default 'ai'   -- 'ai' or 'dm'
```

### `resource_points`
```sql
id                    uuid primary key default uuid_generate_v4()
map_id                uuid references campaign_maps(id) on delete cascade
x_pct                 float not null
y_pct                 float not null
resource_type         text not null
  -- EXTRACTION: 'iron_deposit','copper_deposit','gold_vein','silver_vein',
  --   'gem_cluster','coal_seam','stone_quarry','salt_flat','sulfur_vent'
  -- BIOLOGICAL: 'deep_fishery','coastal_fishery','river_fishery',
  --   'fertile_farmland','grazing_land','orchard','ancient_forest',
  --   'managed_woodland','rare_herbs'
  -- TRADE: 'natural_harbor','river_ford','mountain_pass',
  --   'trade_crossroads','oasis','river_confluence'
  -- SPECIAL: 'arcane_nexus','ancient_ruins','volcanic_soil','hot_springs'
richness              float not null   -- 0.1 to 1.0
name                  text
influence_radius_pct  float default 0.08
placed_by             text default 'ai'
```

### `points_of_interest` (new)
```sql
id              uuid primary key default uuid_generate_v4()
map_id          uuid references campaign_maps(id) on delete cascade
campaign_id     uuid references campaigns(id) on delete cascade
dm_id           uuid references profiles(id)
x_pct           float not null
y_pct           float not null
poi_category    text not null
  -- 'settlement','military','arcane','dungeon',
  -- 'religious','natural','ruin','infrastructure','wilderness'
poi_type        text not null   -- specific type within category, see registry below
name            text            -- DM-given name, e.g. "The Tomb of Aethon"
description     text            -- DM notes, visible only to DM
player_hint     text            -- optional: what players can discover about this place
is_discovered   boolean default false  -- DM can mark when party finds it
is_visible_to_players boolean default false  -- controls player map visibility (future)
-- Link to other entities if the PoI is a full generated location
linked_shop_id  uuid references shops(id)
linked_town_id  uuid references world_towns(id)
-- Optional: uploaded custom image for this PoI (AI or DM-provided)
image_url       text
created_at      timestamptz default now()
```

### `world_towns`

```sql
id                uuid primary key default uuid_generate_v4()

y_pct             float not null

specializations   text[]
trade_partners    jsonb
price_index       jsonb

shop_id           uuid references shops(id)

```

### AI Path Workflow

1. **Configure**: Map size (`region` / `kingdom` / `continent`), map style, biome profile
2. **Generate**: 3 DALL-E 3 images generated in parallel. Show a progress state.
3. **Select**: DM picks one. The others are discarded. `is_selected = true` set on chosen map.
4. **Auto-classify terrain**: GPT-4o Vision runs automatically. DM sees a progress message.
5. **Review terrain**: DM can override any polygon's terrain type.
6. **Auto-place resources**: GPT-4o places resource points. DM sees them appear on the map.
7. **Review resources**: DM can drag, delete, or add resource points.
8. **Mark ready**: `setup_stage = 'ready'`. DM is taken to the main map view.

Regeneration limit: 3 total DALL-E calls per campaign. Counter shown as "X of 3 free generations remaining." Terrain and resource operations are always free.

### Upload Path Workflow

1. **Upload image**: DM drags or selects an image file. Accepted formats: PNG, JPEG, WebP. Max size: 20MB. Image is uploaded to Supabase Storage immediately.

2. **Map metadata**: DM provides:

   - Map style (optional, used only for PoI icon theme)

3. **AI-assist or Manual** — two options shown side by side:

4a. **AI-assist sub-path**:
   - GPT-4o Vision classifies terrain → same review step as AI path

## Resource Point Placement Strategy (for GPT-4o)

Target density by map size:
- `region`: 30–60 points
- `kingdom`: 80–140 points  
- `continent`: 200–350 points

Placement density rules (earth-like):
- **Coastlines**: 1–3 harbor/fishery points per significant coastal stretch
- **Rivers**: fishery every ~5% of river length; ford at crossings; confluence where rivers meet
- **Mountain ranges**: dense iron, stone, and coal clusters; 1–2 gold/silver veins per range
- **River valleys**: highest density fertile_farmland (Nile effect)
- **Plains**: fertile_farmland roughly every 8–12% of plains area
- **Forest**: ancient_forest and woodland clusters; rare_herbs at forest edges
- **Desert**: salt_flat in dry lake beds; oasis at prominent features; gem clusters
- **Volcanic**: sulfur_vent, volcanic_soil, hot_springs

Points should visually cluster in rich areas and thin out in barren ones. Return all points as JSON: `[{ x_pct, y_pct, resource_type, richness, name, influence_radius_pct }]`

## Elevation Model

### Why elevation matters

Elevation is the spine of the geography. It determines climate, travel difficulty, visibility, hazards, and ecosystem — all of which flow into the storytelling layer. Every terrain type has a realistic earth-like elevation range. Elevation is stored per terrain area and interpolated at any (x, y) using the same IDW engine as resources.

### Elevation ranges by terrain type

Store `elevation_min_m` and `elevation_max_m` on each `terrain_areas` record. The midpoint is the base elevation for IDW; noise within the range creates natural variation.

```typescript
export const TERRAIN_ELEVATION_RANGES: Record<string, { min: number; max: number }> = {
  ocean:          { min: 0,    max: 0    },
  deep_sea:       { min: 0,    max: 0    },
  coast:          { min: 0,    max: 15   },  // beach to low bluffs
  river:          { min: 5,    max: 120  },  // rivers cut valleys near sea level
  lake:           { min: 2,    max: 300  },
  plains:         { min: 50,   max: 500  },
  grassland:      { min: 100,  max: 700  },
  farmland:       { min: 10,   max: 350  },  // concentrated in river valleys
  forest:         { min: 50,   max: 1500 },
  deep_forest:    { min: 100,  max: 1200 },
  jungle:         { min: 0,    max: 800  },  // lowland jungles most common
  hills:          { min: 200,  max: 800  },
  highlands:      { min: 600,  max: 1500 },
  mountains:      { min: 800,  max: 3500 },
  high_mountains: { min: 2500, max: 6000 },
  swamp:          { min: 0,    max: 50   },  // always low, near water
  wetlands:       { min: 0,    max: 80   },
  desert:         { min: 100,  max: 1500 },  // varies — Sahara low, Gobi high
  badlands:       { min: 200,  max: 1200 },
  tundra:         { min: 0,    max: 500  },
  arctic:         { min: 0,    max: 2000 },
  volcanic:       { min: 500,  max: 3500 },
};

// Earth rules enforced at terrain classification time:
// 1. Water bodies must always be lower than adjacent non-water terrain.
// 2. Mountain passes get elevation slightly below surrounding mountain midpoint
//    — they exist because they are the LOW point through the range.
// 3. River elevation decreases from source (surrounding terrain elevation)
//    to mouth (0–15m at sea level).
```

### Elevation in the IDW engine

Add `elevation_m` as a first-class dimension alongside resource scores. Computed identically: weighted average of `computed_elevation_m` values from nearby terrain area polygon centroids.

```typescript
export function elevationLabel(m: number): string {
  if (m < 10)   return 'Sea level';
  if (m < 200)  return 'Lowlands';
  if (m < 500)  return 'Uplands';
  if (m < 1000) return 'Highlands';
  if (m < 2000) return 'Alpine';
  if (m < 3500) return 'Subalpine';
  return 'High alpine peak';
}
```

### Contour overlay

A "Show contours" toggle samples the IDW elevation function on an 80×80 grid, runs `d3.contours()`, and draws SVG paths at 200m intervals below 1000m and 500m intervals above. Line color: warm brown at low elevation fading to near-white at peaks. Opacity: 35% so the map image stays readable.

### Climate derivation (`/lib/world/climate.ts`)

Rule-based. No AI.

```typescript
// Rules (applied in order, first match wins):
//   coastal (water_access > 0.7) + elevation < 500m + warm biome  → tropical
//   coastal + elevation < 500m                                      → temperate_maritime
//   elevation > 2000m                                               → highland
//   elevation > 1500m + cold biome                                  → subarctic
//   desert terrain type                                              → arid
//   low elevation, inland                                            → temperate_continental
//
// Temperature at sea level by zone:
//   tropical=32°C, subtropical=28°C, temperate=18°C, subarctic=8°C, arctic=-5°C
// Lapse rate: -6.5°C per 1000m elevation
// Coastal modifier: -4°C summer, +5°C winter
//
// Rainfall by zone:
//   Windward coast: 1200–2000mm   Leeward of mountains: 300–600mm
//   Plains: 500–900mm             Desert: 50–250mm    Tropical: 1500–3000mm
//
// Growing season: months where avg temp > 5°C and rainfall > 20mm/month
// Pass open months (mountains): base 7, minus 1 per 500m above 1500m, min 0
```

## Math Engine

### `/lib/world/resourceInterpolation.ts` — IDW Engine

```typescript
// IDW formula at point (x, y) for dimension d:
//   value(x,y,d) = Σ [richness_i * dim_weight(type_i, d) * (1 / dist_i²)]
//                  ───────────────────────────────────────────────────────
//                  Σ [1 / dist_i²]
//
// Only include points where dist_i < point.influence_radius_pct
// Fall back to terrain area base score when no points are in range.

// RESOURCE_DIMENSIONS: resource_type → { dimension: contribution_weight }
// (full table defined in prompt — see prior version)
```

### `/lib/world/spatialIndex.ts` — R-tree

Build index once on map load. Query with bounding box before distance filter. Must return results in < 1ms for 350 points.

### `/lib/world/terrainScores.ts`, `gravityModel.ts`, `centralPlace.ts`, `priceIndex.ts`

All identical to prior spec. All spatial distances use Euclidean distance between `(x_pct, y_pct)` floats.

### PoI Economic Modifiers

When computing IDW at a point, also include PoI `economicModifiers` from the registry as additional "resource points" — treat them as synthetic resource nodes with their modifier values as richness and a fixed `influence_radius_pct` of `0.05`. This means placing a `bridge` or `aqueduct` PoI genuinely improves the trade and water access scores of nearby town placement candidates.

## Notes for Windsurf

- All coordinates are normalized floats `0.0–1.0`.
- `poiDefinitions.ts` is the single source of truth for all PoI types. The UI, DB, and map renderer all reference it. Never hardcode PoI type strings anywhere else.
- PoI `economicModifiers` are injected into the IDW engine as synthetic resource nodes — this is how PoIs affect town generation economics automatically.
- The map canvas component must work identically for AI-generated and uploaded maps. The `creation_method` field on `campaign_maps` affects workflow pages only — the main map view doesn't branch on it.
- All four workflow pages (AI config, AI selection, upload, manual editor) converge on the same terrain review and resource review pages.
- `resource_snapshot` in `world_towns` is frozen at creation time. It does not change when resource points or PoIs are edited later.
- Unit tests required for all math engine files before any UI work. Use `vitest`.
- `worldBuilder` is idempotent — re-running on same data gives same results.
- All API routes follow the existing `{ data, error }` response contract.
- Add `loading.tsx`, `error.tsx`, `not-found.tsx` to all new route segments.
- Do not call AI endpoints if data already exists in the database.
